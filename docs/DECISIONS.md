# PlayerPulse architectural decisions

Major product, architecture, methodology, security, deployment, and data-rights
decisions are append-only records. Superseded decisions should be marked rather
than silently removed.

## ADR-001: modular monolith with separate web client

- Status: accepted
- Date: 2026-07-20

PlayerPulse uses one FastAPI backend, one React single-page frontend, and one
relational database. Analytics remain cohesive backend modules rather than a
separate service. This keeps the five-day portfolio build testable and avoids
distributed-system overhead while preserving clear module boundaries.

## ADR-002: SQLite demo and PostgreSQL production configuration

- Status: accepted
- Date: 2026-07-20

SQLite is the zero-configuration default for local demonstrations and fast
tests. PostgreSQL is configured for production and CI integration. SQLAlchemy
and Alembic provide the portability boundary. UUID identifiers and UTC
timestamps are the persistence conventions.

## ADR-003: deterministic synthetic public data

- Status: accepted
- Date: 2026-07-20

Tests, CI, public demos, screenshots, and walkthroughs use fictional data
generated by project code with a fixed seed. Synthetic records are explicitly
labelled and must not imitate a real match. Third-party raw data is never a
runtime prerequisite.

## ADR-004: conservative external dataset rights gate

- Status: accepted
- Verified at: 2026-07-20T04:57:02Z

The official Metrica Sports repository and its `Legal stuff` section were
opened and checked. It requests responsible use and public acknowledgement but
does not expose a separate standard root licence. Metrica files are therefore
local-import-only and cannot be committed, packaged, hosted, or redistributed
without explicit verified permission.

The official Hudl StatsBomb Open Data repository and official `LICENSE.pdf`
link were opened and checked. The repository describes public research and
football-analytics use and requires StatsBomb source attribution plus its
current publication logo/branding condition. The optional adapter remains
verification-gated and no dataset or logo is downloaded during Day 1.

Official pages checked:

- https://github.com/metrica-sports/sample-data
- https://github.com/metrica-sports/sample-data#legal-stuff
- https://github.com/hudl/open-data
- https://github.com/hudl/open-data/blob/master/LICENSE.pdf

Terms may change. Exact source commit and terms digest must be captured when an
adapter is enabled in Day 2 and re-verified immediately before release.

## ADR-005: original assets and restrained application identity

- Status: accepted
- Date: 2026-07-20

The frontend will use an original abstract pitch/pulse SVG mark and project
design tokens. It will not include club badges, league marks, provider logos,
player photographs, or protected football imagery. Text links provide source
attribution until current terms require and permit a specific asset.

## ADR-006: non-medical explainability before machine learning

- Status: accepted
- Date: 2026-07-20

The primary Day 3 indicator will be deterministic, transparent, versioned, and
threshold-tested. Score and data/model confidence remain separate. Optional
anomaly detection is disabled below its minimum-data gate and is never framed
as medical validation or injury probability.

## ADR-007: secure local-first processing

- Status: accepted
- Date: 2026-07-20

Public production uploads default to disabled. The MVP accepts only documented
CSV/JSON inputs, never archives or user-supplied pickle/Joblib models. Local
disk is a temporary cache in hosted environments; durable summaries belong in
PostgreSQL. In-process jobs are deliberately non-durable and documented as
such rather than adding a distributed queue.

## ADR-008: main-branch remote workflow

- Status: accepted
- Date: 2026-07-20

The user explicitly authorized pushing PlayerPulse to
`https://github.com/Dippy2003/football-fatigue-analytics.git`. Development stays
on the professional `main` branch; no phase-named branches are created. Daily
phase markers remain annotated tags required by the governing specification.
The remote was confirmed empty before the first push.

## ADR-009: provider-neutral metre schemas and fail-closed adapters

- Status: accepted
- Date: 2026-07-21

All tracking and event adapters normalize into strict provider-neutral schemas
using seconds within period and a 105 by 68 metre pitch. Analytics operate only
on these canonical tables. This prevents provider conventions from leaking into
metric formulas and makes boundary behavior testable.

The synthetic generator is the only source allowed without additional action.
Metrica-compatible imports require developer-supplied local files. StatsBomb
event imports additionally require an explicit current-rights acknowledgement.
Neither adapter downloads, bundles, or commits provider data.

## ADR-010: transparent quality-preserving movement pipeline

- Status: accepted
- Date: 2026-07-21

Cleaning retains evidence: duplicates are reported, short interpolation is
marked, long gaps remain unresolved, and implausible speed observations are
flagged instead of silently removed. The quality score exposes completeness,
timestamp monotonicity, and plausible-speed rate separately from later
performance-risk calculations.

Derived local tables use Parquet with PyArrow. Pickle and Joblib are prohibited
for imported tables because they can execute untrusted serialized code. Public
demo outputs are reproducible, synthetic, generated on demand, and Git-ignored.

## ADR-011: typed relational summaries with deterministic raw regeneration

- Status: accepted
- Date: 2026-07-22

Dataset lineage, teams, players, matches, player-match metrics, assessments, and
processing jobs use typed SQLAlchemy columns, UUID keys, UTC timestamps, and six
reversible Alembic revisions. Public synthetic raw/timeline/heatmap/event tables
are regenerated from the stored seed rather than bloating the relational store.
Queryable summaries and explanations remain durable.

## ADR-012: rule indicator primary, anomaly signal separate

- Status: accepted
- Date: 2026-07-22

`rule-risk-v1` is the primary displayed performance-risk indicator. It requires
three physical factors, 60% coverage, and quality of at least 0.50; otherwise it
returns `insufficient_data`. Score and data/model confidence remain separate.
The Isolation Forest is a separately reported optional signal, refuses fewer
than 50 valid records, and is never interpreted as confirmed fatigue.

## ADR-013: fail-closed uploads and in-process jobs

- Status: accepted
- Date: 2026-07-22

Third-party multipart uploads remain disabled by default. When deliberately
enabled for local use they require an explicit provider/manifest, bounded file
count/size, safe `.csv`/`.json` basenames, and reject archives and serialized
models. Processing jobs persist state but execute in-process; the API visibly
states that a restart can require retry rather than implying durable workers.
